# Incident Playbook

インシデント対応の標準手順書。P1〜P4 の重大度別ステップを定義。

## 重大度定義

| 重大度 | 条件 | 応答時間 | 通知先 |
|--------|------|---------|--------|
| **P1 Critical** | サービス全停止 / 決済不可 / データ損失リスク | 5分以内 | 全エンジニア + 経営層 |
| **P2 High** | 主要機能停止 / エラーレート > 5% | 15分以内 | オンコールエンジニア + マネージャー |
| **P3 Medium** | 一部機能劣化 / エラーレート 1-5% | 1時間以内 | 担当チーム |
| **P4 Low** | パフォーマンス低下のみ / エラーレート < 1% | 翌営業日 | 担当チーム (Slack のみ) |

## P1/P2 対応手順 (詳細)

### Phase 1: 検知・初動 (0〜5分)

```
1. アラート受信
   └─ Datadog アラート / ユーザー報告 / 自動検知

2. 初期確認 (Datadog MCP)
   ├─ query_metrics: エラーレート、レイテンシ、スループット
   ├─ list_logs: 直近5分のエラーログ
   └─ list_events: 直近1時間のデプロイ・設定変更

3. 影響範囲の特定
   ├─ 影響サービス一覧
   ├─ エラーレート (%) 
   ├─ 影響ユーザー数 (推定)
   └─ 収益影響 (あり/なし/不明)

4. 重大度判定 → インシデント宣言
   └─ Slack #incidents チャンネルに投稿
      "🚨 [P1] サービス全停止 — 調査中 @channel"
```

### Phase 2: 通知・体制構築 (5〜15分)

```
P1 の場合:
  ├─ インシデントコマンダーを指名
  ├─ ステータスページを "Investigating" に更新
  ├─ 経営層に一次報告
  └─ 顧客向け通知の準備

P2 の場合:
  ├─ 担当者を明確化 (IC + 調査担当)
  └─ ステータスページを "Investigating" に更新
```

### Phase 3: 調査・仮説立案 (15〜45分)

```
1. タイムライン構築
   ├─ デプロイ履歴と問題発生タイミングを突き合わせ
   ├─ 設定変更の有無を確認
   └─ 依存サービスの状態確認

2. 仮説リスト作成 (優先度順)
   優先度の決め方:
   - タイムライン上の「直前イベント」と最も相関が高いもの
   - 過去の類似インシデントで多かった原因

3. 仮説の検証 (各仮説 5分以内で判定)
   ├─ ログから特定のエラーパターンを検索
   ├─ APM トレースで問題箇所を特定
   └─ メトリクスの相関確認

4. 根本原因の特定
```

### Phase 4: 対処・回復 (45〜90分)

```
対処オプション (優先度順):

1. ロールバック (最速、リスク低)
   条件: 直前のデプロイが原因の場合
   手順: 前バージョンへの即時ロールバック

2. フィーチャーフラグ無効化
   条件: 特定機能が原因の場合
   手順: 問題機能のフラグを OFF

3. スケールアウト (症状緩和)
   条件: 負荷増大が原因の場合
   手順: インスタンス数を一時的に増加

4. 設定修正 + デプロイ
   条件: 設定値の誤りが原因の場合
   リスク: 修正に伴うデプロイリスクあり

5. 依存サービスの切り離し (circuit breaker)
   条件: 外部依存サービスの障害が原因の場合
```

### Phase 5: 解決確認・クローズ (回復後)

```
解決確認チェックリスト:
  ├─ エラーレートが正常範囲に戻ったか？ (< 0.5%)
  ├─ レイテンシが正常範囲に戻ったか？
  ├─ ユーザー報告が止まったか？
  ├─ 全依存サービスが正常か？
  └─ モニターがすべて Green になったか？

クローズ処理:
  ├─ ステータスページを "Resolved" に更新
  ├─ 関係者に解決を通知
  ├─ インシデントタイムラインを記録
  └─ Post-Mortem の日程を設定 (24〜48時間以内)
```

## P3 対応手順

```
1. 担当者をアサイン (当番エンジニア)
2. 根本原因を調査 (1〜2時間)
3. 修正 → 次の定期デプロイで適用
4. 影響ユーザーへの個別対応 (必要な場合)
5. 翌日の定例でレビュー
```

## データ収集コマンド (Datadog MCP)

### エラーレート確認

```
datadog_mcp.query_metrics(
  query="sum:trace.http.request.errors{env:production} by {service}.as_rate() / sum:trace.http.request.hits{env:production} by {service}.as_rate()",
  from="-1h"
)
```

### トップエラーログ確認

```
datadog_mcp.list_logs(
  query="status:error env:production",
  sort="-timestamp",
  limit=50,
  from="-30m"
)
```

### デプロイ・イベント確認

```
datadog_mcp.list_events(
  tags="env:production",
  from="-2h",
  priority="normal"
)
```

### APM トレース確認 (遅いリクエスト)

```
datadog_mcp.query_traces(
  query="env:production @duration:>1000000000",  # 1秒以上
  sort="-duration",
  limit=20
)
```

### インフラメトリクス確認

```
datadog_mcp.query_metrics(
  query="avg:system.cpu.user{env:production} by {host}",
  from="-30m"
)
datadog_mcp.query_metrics(
  query="avg:system.mem.pct_usable{env:production} by {host}",
  from="-30m"
)
```

## インシデント対応のアンチパターン

| アンチパターン | 問題 | 正しい対応 |
|-------------|------|-----------|
| 「様子見」が長すぎる | 影響拡大 | 3分以内に重大度判定、15分以内に対処開始 |
| ロールバックを躊躇する | 被害時間延長 | デプロイが原因なら即ロールバック |
| 仮説なしに広範囲を調査 | 時間の無駄 | タイムラインから最有力仮説を1つ選び、5分で検証 |
| Post-Mortem をスキップ | 再発 | P2 以上は必ず実施 (48時間以内) |
| 個人を責める | 心理的安全性の低下 | 「システム・プロセスの問題」として扱う |
